ADDITION 1 — URL param parsing block (insert after the `lang` line, around line 56):

Find this block:
```
    const lang  = url.searchParams.get('lang')  || 'de';
```

Insert the following block immediately after it (before the next blank line / comment):
```javascript

    // ====== SEC-01: Inbound postMessage Origin Allowlist ======
    // Konfigurierbar via: ?allowOrigins=https://shop.example.com,https://other.example.com
    // Ohne ?allowOrigins=: Same-Origin-Nachrichten werden akzeptiert, Cross-Origin ignoriert.
    const _allowOriginsRaw = url.searchParams.get('allowOrigins');
    const _allowOrigins = _allowOriginsRaw
      ? _allowOriginsRaw.split(',').map(o => o.trim()).filter(Boolean)
      : [];

    function isOriginAllowed(origin) {
      if (origin === location.origin) return true;  // same-origin immer erlaubt
      return _allowOrigins.some(allowed => origin === allowed); // nur exakter Match
    }
```

Rules for this function:
- Use `===` for all comparisons — NEVER indexOf, startsWith, includes, or regex (subdomain bypass risk)
- When `?allowOrigins=` is absent: `_allowOrigins` is `[]`, so `some()` returns false for all cross-origin callers
- Same-origin (e.g., http://localhost when viewed via XAMPP) always passes — backward-compatible

ADDITION 2 — Origin guard at top of message handler (insert as first statement in handler body):

Find this exact handler opening:
```javascript
window.addEventListener('message', (event)=>{
```

Change it to:
```javascript
window.addEventListener('message', (event)=>{
      if (!isOriginAllowed(event.origin)) return; // SEC-01: unbekannte Origin stillschweigend ablehnen

```

Do NOT:
- Post any error message back to the rejected origin (information leak)
- Add the guard anywhere other than the very first line of the handler body
- Change the handler's event parameter name or any other existing line

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| cross-origin → iframe | Any page can hold a reference to the viewer's contentWindow and call postMessage(); viewer cannot verify caller identity without explicit origin check |
| URL param → viewer config | The ?allowOrigins= param is visible in the iframe src URL in DevTools; it is routing config, not a secret |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | window.addEventListener('message') in viewer/index.html | mitigate | isOriginAllowed(event.origin) with exact === check as first guard; unauthorized origins return immediately with no response |
| T-01-02 | Tampering | postMessage handler switch(type) — all camera/visibility/orbit commands | mitigate | Blocked by T-01-01 origin guard; no command executes for unauthorized senders |
| T-01-03 | Information Disclosure | console.log('[host→viewer]', type, payload) at line 731 | mitigate | Guard behind `if (showGui)` — payload only logged when ?gui=1 is present |
| T-01-04 | Information Disclosure | Silent reject (no error postback to rejected origin) | accept | Correct posture — sending an error back confirms viewer presence to attacker; `return;` with no response is intentional |
| T-01-05 | Spoofing | Substring bypass of origin check (indexOf/startsWith/regex) | mitigate | Use `===` only — enforced by acceptance criteria grep check |
| T-01-06 | Information Disclosure | ?allowOrigins= value visible in iframe src URL | accept | Origin is not a secret; it is the embedding page's public address bar value; no encoding needed |
</threat_model>
