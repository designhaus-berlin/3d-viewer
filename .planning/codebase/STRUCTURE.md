## Directory Purposes

**`viewer/`:**
- Purpose: The deployable 3D viewer application
- Contains: Single HTML file with all JavaScript inline (`<script type="module">`)
- Key files: `viewer/index.html` — the only file in this directory; do not split it without careful consideration since the import map relies on relative paths to `../vendor/`

**`host-example/`:**
- Purpose: Reference implementations showing how to embed the viewer
- Contains: Two complete HTML pages demonstrating the raw iframe pattern and the Web Component pattern
- Key files: `host-example/index.html` (production-quality example using DB UX design system), `host-example/webcomponent.html` (minimal Web Component demo)
- Note: These pages import Three.js directly via their own `<script type="importmap">` for their own scripts; the viewer iframe has its own import map

**`embed/`:**
- Purpose: Distributable Web Component for teams that prefer a custom-element interface over raw `<iframe>`
- Contains: `embed/webcomponent.js` — single JS file, no dependencies, load with `<script type="module">`
- Key files: `embed/webcomponent.js`

**`assets/`:**
- Purpose: Test and demo 3D models in GLTF binary (`.glb`) format
- Contains: `ae86.glb` (car model, used for wheel alias testing), `260302_Dresden-Cotta_gesamt.glb` (large city model, used in `host-example/index.html`)
- Note: Assets are referenced by relative URL at runtime via `?model=../assets/...`; they are committed to the repo

**`vendor/`:**
- Purpose: All third-party libraries, fully self-hosted (no CDN at runtime)
- Contains: Three.js core, lil-gui, Three.js addons (OrbitControls, GLTFLoader, DRACOLoader, KTX2Loader), DRACO/Basis WASM decoder files
- Key files: `vendor/three.module.min.js` (imported via importmap alias `"three"`), `vendor/addons/` (imported via importmap alias `"three/addons/"`)
- Generated: No — these are vendored library files, manually placed; not generated by a build tool

**`docs/`:**
- Purpose: Human-readable API reference
- Contains: `docs/api.md` — compact postMessage API cheat-sheet (Host→Viewer commands, Viewer→Host events)
- Note: `README.md` in the project root is the full documentation; `docs/api.md` is the short version

**`.planning/codebase/`:**
- Purpose: GSD codebase analysis documents consumed by `/gsd-plan-phase` and `/gsd-execute-phase`
- Generated: Yes (by GSD tooling)
- Committed: Yes (planning artifacts live in the repo)

## Key File Locations

**Entry Points:**
- `viewer/index.html`: The 3D viewer — load this in an `<iframe>`
- `host-example/index.html`: Full integration reference — open directly in browser via XAMPP
- `host-example/webcomponent.html`: Web Component integration reference

**Configuration (URL Parameters — applied at iframe load time):**
- `?model=path` — GLB model path (relative to viewer)
- `?theme=dark|light` — background color
- `?gui=1` — show debug GUI
- `?anim=0|1` — force animation on/off (overrides `prefers-reduced-motion`)
- `?arcLift=`, `?arcMax=`, `?arcSafe=` — arc camera tuning
- `?camPos=x,y,z`, `?camTar=x,y,z`, `?fov=` — initial camera state
- `?camMinY=`, `?camMaxY=` — world-space Y clamp for camera
- `?canvasHover=0` — disable canvas-native hover highlight

**Core Logic (all inside `viewer/index.html`):**
- Three.js scene setup: lines ~85–115
- Debug GUI: lines ~119–228
- Render loop: lines ~236–251
- Model loading (`loadModel`, `createDemoModel`, `frameObject`): lines ~256–320
- Wheel auto-detection (`detectWheels`, `resolveSelector`): lines ~323–380
- Hover highlight (`_applyHover`, `_clearHover`, `_applyExternalHover`): lines ~387–470
- Tween engine (`tween`, `easeInOutCubic`): lines ~488–517
- Camera functions (`animateCameraTo`, `getModelPivot`, `animateCameraArc`): lines ~519–617
- postMessage dispatcher (`window.addEventListener('message', ...)`): lines ~726–807
- Legacy API functions (`setCamera`, `setVisible`, `focusNode`, `enableOrbit`): lines ~626–667

**API Reference:**
- `README.md`: Full docs — URL params, all commands, events, selector syntax, Debug GUI, Blender workflow
- `docs/api.md`: Compact cheat-sheet for quick reference

## Naming Conventions

**Files:**
- HTML entry points: `index.html` (flat, no suffix variants)
- JavaScript modules: `kebab-case.js` (e.g., `webcomponent.js`)
- Vendor files: original library names preserved (e.g., `OrbitControls.js`, `GLTFLoader.js`)
- Assets: snake_case or descriptive names (e.g., `ae86.glb`, `260302_Dresden-Cotta_gesamt.glb`)

**Directories:**
- Lowercase, hyphen-separated for project dirs: `host-example/`, `vendor/addons/`
- Lowercase, single-word for simple dirs: `viewer/`, `embed/`, `assets/`, `docs/`

**JavaScript (inside `viewer/index.html`):**
- Private/internal functions: `_camelCase` prefix (e.g., `_applyHover`, `_clearHover`, `_hoveredMesh`)
- Config objects: `camelCase` noun + `Cfg` suffix (e.g., `arcCfg`, `animCfg`)
- Three.js object variables: short descriptive names (`renderer`, `scene`, `camera`, `controls`, `root`)
- postMessage types: `camelCase` string literals (e.g., `'animateCamera'`, `'setVisibility'`, `'poi'`)

**CSS (inside HTML files):**
- Layout classes: BEM-like descriptive names (`app-header`, `viewer-wrap`, `sidebar-content`)
- State classes: not used; state is controlled via JS attribute/style mutation

## Where to Add New Code

**New postMessage command (Host → Viewer):**
- Add `case 'newCommandName':` to the `switch(type)` block in `viewer/index.html` (~line 732)
- Implement the handler function in the same file (add near related functions)
- Document in `README.md` under "API: Host → Viewer" and add entry to `docs/api.md`
- If exposed via Web Component: add a typed method to `X3DViewer` in `embed/webcomponent.js`

**New outbound event (Viewer → Host):**
- Call `post({ type: 'newEventName', payload: { ... } })` at the relevant point in `viewer/index.html`
- Document in `README.md` under "Events: Viewer → Host" and add entry to `docs/api.md`
- Handle in `host-example/index.html` `switch(d.type)` as a usage example

**New URL parameter:**
- Read via `url.searchParams.get('paramName')` at the top of the script block in `viewer/index.html` (near other URL param reads, ~lines 53–83)
- Document in `README.md` URL params table

**New test/demo model:**
- Place `.glb` file in `assets/`
- Reference via `?model=../assets/yourfile.glb` in the iframe `src`

**New host integration example:**
- Add a new HTML file to `host-example/`
- Reference `../viewer/index.html` as the iframe `src`
- Use the `send(type, payload)` pattern from `host-example/index.html`

**New vendor library:**
- Place in `vendor/` (or `vendor/addons/` if it's a Three.js addon)
- Add an entry to the `<script type="importmap">` in any HTML file that needs it
- Self-host only — no CDN references at runtime

## Special Directories

**`vendor/`:**
- Purpose: All third-party runtime dependencies
- Generated: No (manually vendored)
- Committed: Yes — intentionally committed so the viewer works without any build step or npm install

**`vendor/addons/libs/draco/`:**
- Purpose: DRACO mesh decompression WASM files (needed by `DRACOLoader`)
- Generated: No
- Committed: Yes
- Path must match the `draco.setDecoderPath('../vendor/addons/libs/draco/')` call in `viewer/index.html`

**`vendor/addons/libs/basis/`:**
- Purpose: Basis/KTX2 texture transcoder WASM files (needed by `KTX2Loader`)
- Generated: No
- Committed: Yes
- Path must match `ktx2.setTranscoderPath('../vendor/addons/libs/basis/')` call in `viewer/index.html`

**`.planning/`:**
- Purpose: GSD orchestration and planning documents
- Generated: Yes (by GSD tooling)
- Committed: Yes
